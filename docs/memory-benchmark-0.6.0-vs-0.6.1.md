# pdfalto 0.6.0 vs 0.6.1 — peak memory on large PDFs

Benchmark of the 0.6.1 memory work against the 0.6.0 release, on a corpus of large
HAL/TEL theses (the pathological cases that motivated the work).

## Method

| | |
|---|---|
| 0.6.0 | tag `v0.6.0` (`8f7a213`), xpdf `ba8ecd6` |
| 0.6.1 | `dd10c1d`, xpdf `eb5f219` |
| invocation | `pdfalto -noImage <pdf> <out>` |
| memory ceiling | `ulimit -v` 5 GB (address space) |
| timeout | 30 min |
| corpus | `kept_for_server_2026-06-27`, 1350 files / 61.7 GB, processed largest-first |
| coverage | 344 files = 47.7 GB (**77% by bytes**); the unprocessed tail is small files (median 11 MB) |

Both binaries were built from git in this repository, so the comparison includes the
xpdf submodule bump that is part of the release.

**Timing is CPU (user+sys), not wall clock.** The run was 4-way parallel on a shared,
loaded machine, so wall time is not comparable between rows; large CPU ratios are
meaningful, small ones are noise. Peak RSS is a per-process metric and is unaffected by
that load — `vmstat` confirmed no swapping during the run.

## Results

| outcome | 0.6.0 | 0.6.1 |
|---|---:|---:|
| OK | 129 | 339 |
| FAIL(abort) | 188 | 1 |
| FAIL(segv) | 26 | 1 |
| FAIL(oom-reported) | 0 | 1 |
| FAIL(damaged) | 2 | 2 |

0.6.0 fails on 62% of these documents at a 5 GB ceiling; 0.6.1 fails on 4.

Summed peak RSS over the 129 documents where **both** succeeded:
**270.9 GB -> 6.6 GB (-97.5%)**.

### Caveats when reading the table

- A `0.6.0 peak` of `4.97` GB is the process hitting the ceiling, not a measured peak.
  The real peak is higher and unknown.
- `?` in the pages column: page counts were added part-way through the run, so the
  largest documents lack them.
- The 4 remaining 0.6.1 failures: two are the same document (`tel-01522934` and a
  duplicate) which still needs more than 5 GB; one is a graceful out-of-memory report
  (exit 98, the `GMemException` handler) where 0.6.0 aborted; one is a genuinely
  damaged PDF that fails identically on both.
- The two byte-identical copies of `tel-01522934` failed *differently* (abort vs segv),
  so allocation failure does not always reach the clean exit-98 path.
